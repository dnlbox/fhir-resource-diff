# Spec 42 — Period.start / Period.end date format validation

## Goal

The `fhir-date-format` rule validates named date fields like `birthDate`, `effectiveDateTime`,
`onsetDateTime`, and `recordedDate` — but it does not validate the `start` and `end` fields
inside FHIR Period objects. Common Period fields that hold dates are:

- `effectivePeriod.start` / `.end` (Observation, MedicationStatement)
- `onsetPeriod.start` / `.end` (Condition, AllergyIntolerance)
- `performedPeriod.start` / `.end` (Procedure)
- `period.start` / `.end` (Encounter, CarePlan, Coverage)

When `effectivePeriod.start = "not-a-date"` is generated by the `malformed-date` fault,
the validator reports the resource as **valid** with no warning. This closes the last gap
in fault injection coverage for `malformed-date`:

```bash
fhir-test-data generate medication-statement --faults malformed-date | fhir-resource-diff validate -
# → valid   ← incorrect, effectivePeriod.start contains "not-a-date"
```

## Key interfaces / signatures

The current rule uses a field-name heuristic to find date values (fields ending in `Date`,
`DateTime`, specific known names). Extend the heuristic to also descend into any object whose
parent key ends with `Period` and check its `start` and `end` values:

```typescript
// When walking a resource and encountering a field whose key ends with "Period":
function validatePeriod(period: unknown, path: string, errors: ValidationError[]): void {
  if (!period || typeof period !== 'object') return;
  const p = period as Record<string, unknown>;

  if (typeof p.start === 'string') {
    const err = validateDateOrDateTime(p.start, `${path}.start`);
    if (err) errors.push(err);
  }
  if (typeof p.end === 'string') {
    const err = validateDateOrDateTime(p.end, `${path}.end`);
    if (err) errors.push(err);
  }
}
```

The existing `validateDateOrDateTime` (or equivalent) already handles both `date` and
`dateTime` format variants — reuse it. FHIR `Period.start` and `Period.end` are always
`dateTime` type per the spec, but `date` partial values are also valid in practice.

## Implementation notes

- The key pattern to match: any key ending in `Period` whose value is an object with
  `start` and/or `end` string fields
- Only validate `start`/`end` — do not recurse further inside Period objects
- This is a FORMAT_RULE — runs without a FHIR version flag
- Common field names affected: `effectivePeriod`, `onsetPeriod`, `performedPeriod`,
  `abatementPeriod`, `period`, `scheduledPeriod`, `valuePeriod`, `availabilityExceptions`
- The `period` key (without type prefix, used in Encounter and others) should also match —
  check if the value has `start` / `end` string fields regardless of key name, or specifically
  match the `period` key in addition to `*Period` suffix

## Acceptance criteria

```bash
# effectivePeriod.start bad date must be caught
echo '{"resourceType":"Observation","id":"t1","status":"final","code":{"coding":[{"system":"http://loinc.org","code":"718-7"}]},"subject":{"reference":"Patient/p1"},"effectivePeriod":{"start":"not-a-date"}}' \
  | fhir-resource-diff validate -
# Expected: valid (with warnings) — invalid date at effectivePeriod.start

# onsetPeriod.start bad date must be caught
echo '{"resourceType":"Condition","id":"t1","subject":{"reference":"Patient/p1"},"onsetPeriod":{"start":"not-a-date"}}' \
  | fhir-resource-diff validate -
# Expected: valid (with warnings) — invalid date at onsetPeriod.start

# malformed-date fault on MedicationStatement now fully round-trips
fhir-test-data generate medication-statement --locale us --seed 1 --no-pretty --faults malformed-date \
  | fhir-resource-diff validate - --fhir-version R4
# Expected: valid (with warnings) — invalid date at effectivePeriod.start

# Valid Period dates must pass
echo '{"resourceType":"Observation","id":"t1","status":"final","code":{"coding":[{"system":"http://loinc.org","code":"718-7"}]},"subject":{"reference":"Patient/p1"},"effectivePeriod":{"start":"2024-01-01","end":"2024-12-31"}}' \
  | fhir-resource-diff validate -
# Expected: valid (no Period date warnings)

# Period with missing start/end must pass (fields are optional in Period)
echo '{"resourceType":"Observation","id":"t1","status":"final","code":{"coding":[{"system":"http://loinc.org","code":"718-7"}]},"subject":{"reference":"Patient/p1"},"effectivePeriod":{}}' \
  | fhir-resource-diff validate -
# Expected: valid
```

## Do not do

- Do not validate `Period.id` or `Period.extension` fields
- Do not add Period validation for non-date child fields (only `start` and `end`)
- Do not change the severity — warning, same as other format rule violations
